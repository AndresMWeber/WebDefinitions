# Web Concepts and Acronyms
<p>
WC3 - The World Wide Web Consortium (W3C) is an international community that develops open standards to ensure the long-term growth of the Web.
  
TLD - Top-level domain refers to the last segment of a domain name, or the part that follows immediately after the "dot" symbol.
  
DOM - The W3C Document Object Model (DOM) is a platform and language-neutral interface that allows programs and scripts to dynamically access and update the content, structure, and style of a document.
  
BOM - The Browser Object Model consists of the objects navigator, history, screen, location and document which are children of window.

CRUD - these operations are the four basic operations of persisting data (Create, Read, Update, Delete)
  
AJAX - short for asynchronous JavaScript and XML.  It is a set of web development techniques using many web technologies on the client side to create asynchronous web applications
  
BSON - Binary JSON (data format that MongoDB uses)
  
ODM -  Is an abstraction on top of our database connection: ODMs translate data from documents in our database, to objects in JavaScript.
  
SSL - SSL provides a secure channel between two machines or devices operating over the internet or an internal network.
  
TLS - Transport Layer Security (TLS) is a protocol that provides authentication, privacy, and data integrity between two communicating computer applications. 
  
HTTPS - Hypertext Transfer Protocol Secure (HTTPS) is a variant of the standard web transfer protocol (HTTP) that adds a layer of security on the data in transit through a secure socket layer (SSL) or transport layer security (TLS) protocol connection.
  
REST - REST (REpresentational State Transfer) is an architectural style for developing web services. REST is popular due to its simplicity and the fact that it builds upon existing systems and features of the internet's Hypertext Transfer Protocol (HTTP) in order to achieve its objectives, as opposed to creating new standards, frameworks and technologies.
  
RESTful - A RESTful API -- also referred to as a RESTful web service -- is based on representational state transfer (REST) technology, an architectural style and approach to communications often used in web services development.
</p>

<h1 class="c38" id="h.jsoniu7o8m2k"><span class="c34 c7">Programming Terms/Concepts</span></h1>
<p class="c19"><span class="c1">Primitive:</span><span class="c2">&nbsp;A primitive (primitive value, primitive
  data type) is data that is not an object and has no methods. In JavaScript, there are 6 primitive data
  types: string, number, boolean, null, undefined, symbol (new in ECMAScript 2015).</span></p>
<p class="c19"><span class="c1">Function Expression </span><span class="c2">- Storing an anonymous function in a
  variable.</span></p>
<p class="c19"><span class="c1">Function Statement (Function Literals) </span><span class="c2">- Traditional
  function definition with the function keyword.</span></p>
<p class="c19"><span class="c1">Callback Function</span><span class="c2">&nbsp;- A callback is a function that
  is to be executed after another function has finished executing — hence the name ‘call back’.</span></p>
<p class="c19"><span class="c1">Shadowing </span><span class="c2">- In computer programming, variable shadowing
  occurs when a variable declared within a certain scope has the same name as a variable declared in an
  outer scope:</span></p>
<p class="c19"><span class="c1">Closure </span><span class="c2">- A closure is the combination of a function and
  the lexical environment within which that function was declared. &nbsp;Sometimes represented as an
  expression (usually a function) that can have free variables together with an environment that binds
  those variables (it “closes” the expression).</span></p>
<p class="c19"><span class="c1">Static Properties</span><span class="c2">&nbsp;- properties of a class, or dot
  notation accessible variables stored at the class level and not on the instance level.</span></p>
<p class="c19"><span class="c1">Black Box Programming </span><span class="c2">- Programming in a black-box
  paradigm mainly consists of having functions or methods that hides the inner workings from the user
  (programmer). These functions take a set number of parameters and produce the given output without the
  developer having to worry about how the actual logic is implemented.</span></p>
<p class="c19"><span class="c1">Block Scope</span><span class="c6 c37">&nbsp;</span><span class="c2">- Lexical
  scope that is contained between two curly braces.</span></p>
<p class="c19"><span class="c1">Function Scope</span><span class="c37 c6">&nbsp;</span><span class="c2">-
  Lexical scope that is contained within a function.</span></p>
<p class="c19"><span class="c1">Lexical Scoping</span><span class="c2">&nbsp;- is a convention used with many
  programming languages that sets the scope (range of functionality) of a variable so that it may only be
  called (referenced) from within the block of code in which it is defined.</span></p>

</ul>
<p class="c19"><span class="c1">Scaffolding </span><span class="c2">- Scaffolding, as used in computing, refers
  to one of two techniques: The first is a code generation technique related to database access in some
  model–view–controller frameworks; the second is a project generation technique supported by various
  tools.</span></p>
<p class="c19"><span class="c1">Bootstrapping </span><span class="c2">- An example of bootstrapping is in some
  web frameworks. You call index.php (the bootstrapper), and then it loads the frameworks helpers, models,
  configuration, and then loads the controller and passes off control to it.</span></p>
<p class="c19"><span class="c1">Tree Shaking</span><span class="c2">&nbsp;- Tree shaking means removing “dead
  code” from the bundle you ship to your users. If you add some code that you never use in your import
  statements, that’s not going to be sent to the users of your app, to reduce file size and loading
  time.</span></p>
<p class="c19"><span class="c1">Transpilation </span><span class="c2">- Transpilers, or source-to-source
  compilers, are tools that read source code written in one programming language, and produce the
  equivalent code in another language.</span></p>
<p class="c19"><span class="c1">Boilerplate </span><span class="c2">- A boilerplate is a unit of writing that
  can be reused over and over without change. By extension, the idea is sometimes applied to reusable
  programming, as in “boilerplate code.”</span></p>
<p class="c19"><span class="c1">Curried (Currying a) Function </span><span class="c2">- Currying is a process in
  functional programming in which we can transform a function with multiple arguments into a sequence of
  nested functions. It returns a new function that expects the next argument inline.</span></p>
<p class="c19"><span class="c1">Higher Order (x) </span><span class="c6">- Higher-order programming is a style
  of computer programming that uses software components, like functions, modules or objects,</span><span
  class="c7 c6">&nbsp;as values</span><span class="c2">. It is usually instantiated with, or borrowed
  from, models of computation such as lambda calculus which makes heavy use of higher-order functions.
  &nbsp;Functions that operate on other functions, either by taking them as arguments or by returning
  them, are called higher-order functions.</span></p>
<p class="c19"><span class="c1">First Class Citizen </span><span class="c2">- is an entity which supports all
  the operations generally available to other entities. These operations typically include being passed as
  an argument, returned from a function, modified, and assigned to a variable. (e.g. functions are first
  class citizens in JavaScript)</span></p>
<p class="c19"><span class="c1">Prototype Delegation</span><span class="c6">&nbsp;- herein an object will search
  its entire prototype chain for a given property (for example, a Chicken object delegating a layEgg
  behavior to a </span><span class="c6">higher-up prototype</span><span class="c2">&nbsp;Bird
  object.)</span></p>
<p class="c19"><span class="c1">Object Linked to Other Object (OLOO) Pattern</span><span class="c2">&nbsp;- In
  this pattern, we don’t use a function to create objects at all. Instead, we define a blueprint object of
  sorts, which we then explicitly use as the prototype for any individual objects we need. In JavaScript
  this is implemented like so: Object.create(myObj). &nbsp;OLOO allows us to share like behaviors and
  check the type of individual objects, all while sidestepping the class illusions inherent in the
  constructor and pseudo-classical patterns.</span></p>

  programming is a programming paradigm — a style of building the structure and elements of computer
  programs—that expresses the logic of a computation without describing its control flow.</span></p>
<p class="c19"><span class="c1">MVC Pattern</span><span class="c7 c6">&nbsp;</span><span class="c6">- &nbsp;To

  class="c2">(Model-View-Controller) that adds the Model in such a way that:</span></p>

    made on the server, a </span><span class="c7 c6">Controller </span><span class="c2">handles
    it.</span></li>
<li class="c9"><span class="c6">This </span><span class="c7 c6">Controller </span><span class="c6">will
    communicate with </span><span class="c7 c6">Models</span><span class="c2">.</span></li>
<li class="c9"><span class="c7 c6">Models </span><span class="c2">will read and write data directly to a
    database.</span></li>
<li class="c9"><span class="c6">When the </span><span class="c7 c6">Controller </span><span class="c6">has
    all information, it can pass that data to the </span><span class="c7 c6">View</span><span
    class="c2">.</span></li>
<li class="c9"><span class="c6">The </span><span class="c7 c6">View </span><span class="c6">generates a
  </span><span class="c7 c6">Response </span><span class="c6">which is a </span><span class="c7 c6">HTML
  </span><span class="c2">page rendered to a client (in a browser).</span></li>
</ul>
<p class="c19"><span class="c1">Programming Paradigm</span><span class="c2">&nbsp;- A programming paradigm is a
  fundamental style of building the structure and elements of a program. The styles and capabilities of
  programming languages are defined by their paradigms.</span></p>
<h2 class="c57 c41" id="h.l6yrq7wq8uu8"><span class="c28 c7 c84">Common Programming Paradigms</span></h2>
<ul class="c39 lst-kix_p7h11iu7c5nl-0 start">
<li class="c15"><span class="c1">Imperative: </span><span class="c2">Programming with an explicit sequence
    of commands that update state.</span></li>
<li class="c15"><span class="c1">Declarative: </span><span class="c2">Programming by specifying the result
    you want, not how to get it.</span></li>
<li class="c15"><span class="c1">Structured:</span><span class="c2">&nbsp;Programming with clean, goto-free,
    nested control structures.</span></li>
<li class="c15"><span class="c1">Procedural: </span><span class="c2">Imperative programming with procedure
    calls.</span></li>
<li class="c15"><span class="c1">Functional (Applicative)</span><span class="c2">: Programming with function
    calls that avoid any global state.</span></li>
<li class="c15"><span class="c1">Function-Level (Combinator): </span><span class="c2">Programming with no
    variables at all.</span></li>
<li class="c15"><span class="c1">Object-Oriented: </span><span class="c2">Programming by defining objects
    that send messages to each other. Objects have their own internal (encapsulated) state and public
    interfaces. Object orientation can be:</span></li>
<li class="c15"><span class="c1">Class-based: </span><span class="c2">Objects get state and behavior based
    on membership in a class.</span></li>
<li class="c15"><span class="c1">Prototype-based: </span><span class="c2">Objects get behavior from a
    prototype object.</span></li>
<li class="c15"><span class="c1">Event-Driven: </span><span class="c2">Programming with emitters and
    listeners of asynchronous actions.</span></li>
<li class="c15"><span class="c1">Flow-Driven: </span><span class="c2">Programming processes communicating
    with each other over predefined channels.</span></li>
<li class="c15"><span class="c1">Logic (Rule-based): </span><span class="c2">Programming by specifying a set
    of facts and rules. An engine infers the answers to questions.</span></li>
<li class="c15"><span class="c1">Constraint:</span><span class="c2">&nbsp;Programming by specifying a set of
    constraints. An engine finds the values that meet the constraints.</span></li>
<li class="c15"><span class="c1">Aspect-Oriented: </span><span class="c2">Programming cross-cutting concerns
    applied transparently.</span></li>
<li class="c15"><span class="c1">Reflective: </span><span class="c2">Programming by manipulating the program
    elements themselves.</span></li>
<li class="c15"><span class="c1">Array: </span><span class="c2">Programming with powerful array operators
    that usually make loops unnecessary.</span></li>
</ul>
<p class="c35"><span class="c1">Design Pattern</span><span class="c2">&nbsp;- a design pattern is a tried and
  tested solution to a common programming pattern. It could be considered a best practice. If you approach
  a program using an Object Oriented paradigm, there are a number of design patterns you can then draw on
  to solve specific problems.</span></p>
<p class="c35 c32"><span class="c2"></span></p>
<h3 class="c38 c77" id="h.cgyg7p18suwu"><span class="c29"><a class="c21"
    href="https://www.google.com/url?q=https://sourcemaking.com/design_patterns&amp;sa=D&amp;ust=1572995788401000">Design
    Pattern Classifications:</a></span></h3>
<p class="c35"><span class="c1">Creational Design Patterns </span><span class="c2">- These design patterns are
  all about class instantiation. This pattern can be further divided into class-creation patterns and
  object-creational patterns. While class-creation patterns use inheritance effectively in the
  instantiation process, object-creation patterns use delegation effectively to get the job done.</span>
</p>
<p class="c35"><span class="c1">Structural Design Patterns </span><span class="c2">- These design patterns are
  all about Class and Object composition. Structural class-creation patterns use inheritance to compose
  interfaces. Structural object-patterns define ways to compose objects to obtain new
  functionality.</span></p>
<p class="c35"><span class="c1">Behavioral Design Patterns </span><span class="c2">- These design patterns are
  all about Class's objects communication. Behavioral patterns are those patterns that are most
  specifically concerned with communication between objects.</span></p>
<p class="c32 c35"><span class="c2"></span></p>
<h3 class="c38 c77" id="h.ahl3uovw2dag"><span class="c29"><a class="c21"
    href="https://www.google.com/url?q=https://airbrake.io/blog/design-patterns/software-design-patterns-guide&amp;sa=D&amp;ust=1572995788402000">Common
    Design Patterns</a></span></h3>
<p class="c35"><span class="c1">Singleton </span><span class="c2">- Only one instance of the given class may
  exist at any time.</span></p>
<p class="c35"><span class="c1">Strategy </span><span class="c2">- A local strategy will change based on a given
  situation.</span></p>
<p class="c35"><span class="c1">Factory </span><span class="c2">- You can create many different objects from one
  Factory based on criteria without the need to directly instantiate the class.</span></p>
<p class="c35"><span class="c1">Builder </span><span class="c2">- Splits up the construction of an object from
  the representation almost like an interface to the object so that we can customize the representation of
  that object.</span></p>
<p class="c35"><span class="c1">Chain of Responsibility </span><span class="c2">- A stack type implementation
  that checks the objects as it goes for validity in order to continue, often used as a failsafe.</span>
</p>
<p class="c35"><span class="c1">Template</span><span class="c2">&nbsp;- create a “template” for high order
  operations/order usually as an abstract class and then the class that extends that will fill in the
  details with helper functions or implementing those sub-functions of the high order operation.</span>
</p>
<p class="c35"><span class="c1">State </span><span class="c2">- Allows behavior of the class to change based on
  the current state of the object.</span></p>

<li class="c20"><span class="c16 c76">avoid variable hoisting from within blocks</span></li>
<li class="c40 c52 c71"><span class="c16 c76">protect against polluting the global environment and</span>

<p class="c19"><span class="c1">Rest Pattern</span><span class="c6">&nbsp;(</span><span class="c1">ES6)
</span><span class="c2">- allows you to spread as many components of a container into variables and catch
  the “rest” of the container’s components you did not specify variables for.</span></p>
<p class="c19"><span class="c1">Spread (ES6) </span><span class="c2">- allows you to expand something that is
  currently grouped inside a particular container and assign it to a different container</span></p>
<p class="c19"><span class="c1">for/in Loop (ES 1st Edition) </span><span class="c2">- When using for/in loops
  with arrays, it is important to note that the “key” of an array is just the index. So when using a
  for/in loop with an array you will not get the value at each index, instead, you will get the
  index.</span></p>
<p class="c19"><span class="c1">for/of Loop (ES6)</span><span class="c2">&nbsp;- a recent addition that is
  specifically intended to iterate over iterable objects, like arrays, strings, maps, and sets.
  Syntactically, it is almost identical to the for/in loop except you replace in with of. Functionally
  they are different. Instead of getting the key, like the for/in loop, you get the actual value at that
  location.</span></p>
<p class="c18"><span class="c1">Destructuring Assignment </span><span class="c2">- Taking an array/object and
  splitting its component indices and storing into receiving variables. </span></p>
<p class="c18 c52"><span class="c6 c81">const</span><span class="c45 c6">&nbsp;[</span><span
  class="c6 c88">first</span><span class="c45 c6">, </span><span class="c88 c6">second</span><span
  class="c6 c45">,</span><span class="c6 c92">&nbsp;...rest</span><span class="c45 c6">] = </span><span
  class="c6 c73">['some', 'long', 'array', 'of', 'values'</span><span class="c45 c6">]</span></p>
<p class="c19"><span class="c1">Anonymous closure </span><span class="c6">- these functions have its own
  evaluation environment or “closure”, and then we immediately evaluate it. This lets us hide variables
  from the parent (global) namespace.</span></p>
<p class="c19"><span class="c1">Module pattern </span><span class="c2">- was used to mimic the concept of
  classes (since JavaScript doesn’t natively support classes) so that we can store both public and private
  methods and variables inside a single object.</span></p>
<p class="c19"><span class="c1">Hoisting </span><span class="c2">- Hoisting is JavaScript's default behavior of
  moving all declarations to the top of the current scope (to the top of the current script or the current
  function). &nbsp;JavaScript only hoists declarations, not initializations.</span></p>
<p class="c19"><span class="c1">Promise (ES2015) </span><span class="c2">- A Promise is an object representing
  the eventual completion or failure of an asynchronous operation. Essentially, a promise is a returned
  object to which you attach callbacks, instead of passing callbacks into a function.</span></p>
<p class="c19"><span class="c1">Arrow Function (ES6)</span><span class="c2">&nbsp;- These are anonymous
  functions with their own special syntax that accept a fixed number of arguments, and operate in the
  context of their enclosing scope — ie the function or other code where they are defined.</span></p>
<p class="c19"><span class="c1">Node Require vs Import (ES2015)</span><span class="c2">&nbsp;- Require can be
  called within a scope, but import can only be called in the global scope, therefore reducing a lot of
  pressure on transpilers that cannot introspect/tree shake properly in that situation.</span></p>
<p class="c19"><span class="c1">Polyfill </span><span class="c2">- In web development, a polyfill is code that
  implements a feature on web browsers that do not support the feature. Most often, it refers to a
  JavaScript library that implements an HTML5 web standard, either an established standard (supported by
  some browsers) on older browsers, or a proposed standard (not supported by any browsers) on existing
  browsers. Formally, "a polyfill is a shim for a browser API"</span></p>
<p class="c19"><span class="c1">Strict Mode</span><span class="c2">&nbsp;- JavaScript's strict mode, introduced
  in ECMAScript 5, is a way to opt in to a restricted variant of JavaScript, thereby implicitly opting-out
  of "sloppy mode". </span></p>
<ul class="c39 lst-kix_k70gdmt4gn0r-0 start">
<li class="c0"><span class="c2">Strict mode eliminates some JavaScript silent errors by changing them to
    throw errors.</span></li>
<li class="c0"><span class="c2">Strict mode fixes mistakes that make it difficult for JavaScript engines to
    perform optimizations: strict mode code can sometimes be made to run faster than identical code
    that’s not strict mode.</span></li>
<li class="c0"><span class="c2">Strict mode prohibits some syntax likely to be defined in future versions of
    ECMAScript.</span></li>
<li class="c0"><span class="c2">It prevents, or throws errors, when relatively “unsafe” actions are taken
    (such as gaining access to the global object).</span></li>
<li class="c0"><span class="c2">It disables features that are confusing or poorly thought out.</span></li>
<li class="c0"><span class="c2">Strict mode makes it easier to write “secure” JavaScript.</span></li>
</ul>

    UI will look at that particular time.</span></li>
<li class="c0"><span class="c7 c6">Component Based -</span><span class="c2">&nbsp;Conceptually, components

  </span><span class="c7 c6">JavaScript/React </span><span class="c6">code. The syntax is intended to be
    used by preprocessors (i.e., transpilers like </span><span class="c7 c6">Babel</span><span
    class="c6">) to transform </span><span class="c7 c6">HTML</span><span class="c2">-like text found in
    JavaScript files into standard JavaScript objects that a JavaScript engine will parse.</span></li>
<li class="c0"><span class="c7 c6">React Native</span><span class="c6">&nbsp;- React Native is a framework
    for building native applications using JavaScript. React Native compiles to native app components,
    which makes it possible for you to build native mobile applications. In React JS, React is the base
    abstraction of </span><span class="c7 c6">React DOM </span><span class="c2">for the web platform,
    while with React Native, React is still the base abstraction but of React Native. So the syntax and
    workflow remain similar, but the components are different.</span></li>

  style="overflow: hidden; display: inline-block; margin: 0.00px 0.00px; border: 0.00px solid #000000; transform: rotate(0.00rad) translateZ(0px); -webkit-transform: rotate(0.00rad) translateZ(0px); width: 361.82px; height: 236.50px;"><img

<ul class="c39 lst-kix_4x54hfs9yslp-0 start">
<li class="c5"><span class="c7 c6">Action Creator</span><span class="c2">&nbsp;- Changes the state of the
    app: produces an object.</span></li>
<li class="c5"><span class="c7 c6">Action </span><span class="c2">- The action object that will run after
    action has been created from template Action Creator.</span></li>
<li class="c5"><span class="c7 c6">Dispatch</span><span class="c2">&nbsp;- Receives any action and feeds to
    reducers.</span></li>
<li class="c5"><span class="c7 c6">Reducers</span><span class="c2">&nbsp;- Process forms/data that comes in,
    modify and return processed data</span></li>
<li class="c5"><span class="c7 c6">State</span><span class="c2">&nbsp;- Receives reducer data that gets
    added to a state object which waits until new actions to update next.</span></li>
</ul>
<p class="c35 c63"><span class="c28 c7 c36">React-Redux</span><span
  style="overflow: hidden; display: inline-block; margin: 0.00px 0.00px; border: 0.00px solid #000000; transform: rotate(0.00rad) translateZ(0px); -webkit-transform: rotate(0.00rad) translateZ(0px); width: 254.73px; height: 172.50px;"><img

    title=""></span></p>
<ul class="c39 lst-kix_vllfncr2r0km-0 start">
<li class="c5"><span class="c7 c6">Provider </span><span class="c2">- takes in the redux store/state as a
    prop at the top level around the App.</span></li>
<li class="c5"><span class="c7 c6">Connect </span><span class="c2">- registers action creators and hooks
    them up with the Provider at the top level via the context system to allow the actions to affect the
    state. &nbsp;Connect also is able to hook up Components with the prodiver’s redux state and also
    remap any redux state values to different prop names. </span></li>
</ul>
<p class="c35 c63"><span class="c1">Async/Sync Action Creators </span><span class="c2">- disallowed in redux
  actions: this is due to post-transpilation code returning a request object, not the plain object we are
  expecting. &nbsp;You need middleware!</span></p>
<p class="c35 c63"><span class="c7 c6">Redux Thunk</span><span class="c2">&nbsp;- Middleware solution for
  asynchronous action creators to allow pausing of actions (enforcing synchronicity) between
  dispatch/reducer cycle and remove race conditions.</span></p>
<h3 class="c35 c63 c77" id="h.yavl831a3e5q"><span class="c50 c47 c7">Rules of Reducers:</span></h3>
<ol class="c39 lst-kix_z0ur0j81a84n-0 start" start="1">
<li class="c5"><span class="c6">Must return </span><span class="c37 c6">any</span><span
    class="c2">&nbsp;value besides ‘undefined’</span></li>
<li class="c5"><span class="c6">Produces ‘state’, or data to be used inside of your app using only previous
    state and the action (reducers are pure)</span><span class="c28 c7 c6">&nbsp;(NO API REQUESTS, USER
    INPUT OR DOM QUERYING etc)</span></li>
<li class="c5"><span class="c2">Must not reach ‘out of itself’ to decide what value to return</span></li>
<li class="c5"><span class="c2">Must not mutate its input ‘state’ argument</span></li>
</ol>

<h3 class="c19 c77" id="h.ssuqs0za33ye"><span class="c7 c37">DNS Types</span><span class="c50 c47 c7">:</span>
</h3>
<p class="c19"><span class="c1">CNAME </span><span class="c6">- maps a subdomain to another domain name.
  &nbsp;This can be another subdomain on the same site (like forwarding www to the root (@)). &nbsp;Once
  you define a </span><span class="c7 c6">CNAME </span><span class="c6">record for a subdomain (host), you
  CAN'T DEFINE another record for that same subdomain. &nbsp;Because of this, you can't use </span><span
  class="c6 c7">CNAME </span><span class="c2">at the root level (where you need other records to exist).
  &nbsp;CANNOT USE THIS ON THE ROOT (@)</span></p>
<p class="c19"><span class="c2">Ex: www.example.com =&gt; example.com</span></p>
<p class="c19"><span class="c1">A/AAAA </span><span class="c2">- maps a subdomain to an IPv4 address. This is
  most commonly used at the root, so it tells your browser where example.com lives. &nbsp;CAN DO THIS ON
  THE ROOT (@). If your domain does not have a static IP you cannot use A record.</span></p>
<p class="c19"><span class="c2">Ex: example.com =&gt; 127.0.0.1</span></p>
<p class="c19"><span class="c2">Ex: example.com =&gt; ::1 </span></p>
<p class="c19"><span class="c1">ALIAS/ANAME/Virtual CNAME- </span><span class="c2">ALIAS is like CNAME, but it's
  a non-standard dns type. &nbsp;So you can use ALIAS to forward your root domain (example.com) to your
  app's server (example.herokudns.com)</span></p>
<p class="c19"><span class="c2">Ex: example.com =&gt; example.netlify.com </span></p>
<p class="c19"><span class="c1">TXT </span><span class="c2">- TXT records, which provide extra info to any
  computer reading the DNS record. &nbsp;TXT records are often used to prove you own a domain - because
  only the domain owner can set a DNS record.</span></p>
<p class="c19"><span class="c2">Ex: @ =&gt; my-domain-is-awesome-123 </span></p>
